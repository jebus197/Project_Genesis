# Genesis UX Brief v2

**Date:** 2026-02-27
**Status:** Canonical design brief (replaces 2026-02-16 version)

## 1) Core Position

Genesis is the world's first **anti-social network**.

That does not mean "no network." It means:
- keep the network functions that survive severe testing,
- remove the social mechanics that repeatedly fail at scale.

This brief adopts a Popperian falsification standard:
- severe testing over comforting assumptions,
- corroboration over confirmation,
- negative data (what failed) treated as first-class knowledge.

## 3) Design Doctrine

Every design element must pass one of two checks:

1. **Corroborated function check:**
   What useful network function does this element implement that survived severe testing?

2. **Falsification check:**
   Which known harmful social mechanic does this element prevent or eliminate?

If neither check is satisfied, the element should not ship.

## 4) What Stays vs What Goes

### 4.1 Corroborated Elements (Must Be Present)

- Structured interaction around missions
- Transparent trust-bearing profiles
- Verifiable outcome history
- Explainable ordering and matching ("Why this appears")
- Public audit and verification paths
- Constitutional governance visibility
- Human/machine distinction with equal dignity and different authority

### 4.2 Falsified Elements (Must Be Absent)

- Popularity ranking
- Network effects as lock-in advantage
- Prestige-weighted visibility
- Engagement bait loops
- Opaque algorithmic ordering
- Earnings gamification
- Pay-for-visibility surfaces

## 5) Primary UX Objective

This is no longer an "explain Genesis" brief.

It is a **show Genesis working** brief.

A first-time visitor must be able to:
1. Understand why the problem matters,
2. See the system operating,
3. Verify key claims independently.

## 6) Experience Sequence (Landing + Mission Board First)

### Stage A: Hook

Open with the stakes (capability is not trustworthiness).  
Use direct, human language. No hype, no slogan spam.

### Stage B: Show

Demonstrate active network behavior:
- mission listings with stake and commission visible,
- matching/order rationale visible,
- trust evidence visible,
- anti-social eliminations visible in practice.

### Stage C: Prove

Show verification paths:
- event-level audit evidence,
- constitutional commitment visibility,
- blockchain record references where relevant.

## 8) Non-Negotiable UX Rules

1. All ranking/matching views must expose ordering logic on-screen.
2. Every major claim must expose an actionable verification path.
3. Mission cards must show staked reward and commission clearly.
4. Trust views must be evidence-first, not vanity-first.
5. Human and machine must be distinguishable, equal dignity, non-identical authority.
6. No featured/trending/viral modules.
7. No social dopamine mechanics (likes, streaks, bait notifications).
8. No paid amplification surfaces.
9. PoC status must remain explicit until First Light.
10. Anti-social absences must be explicit where useful (state what is excluded and why).

## 9) Tone and Voice

Target tone:
- serious, clear, and human
- institutional confidence without corporate stiffness
- honest about limits

Avoid:
- startup cheerleading
- manifesto-only pages with no operational proof
- decorative futurism with no evidence path

Design language target:
- "library-grade trustworthiness"
- "show, then explain"
- "prove, don't imply"

## 10) Acceptance Criteria (Pass/Fail)

A design pass is accepted only if all conditions are met:

1. A new visitor can explain what Genesis is within 10 seconds.
2. A user can explain why a mission appears in their feed.
3. A user can identify at least one independent verification path.
4. A user can identify at least three deliberately eliminated social mechanics.
5. A reviewer can trace trust display -> evidence -> audit event path.
6. Constitutional constraints are preserved (verified by CC review).
7. Narrative coherence and emotional register are preserved (stress-tested by CW).

## 11) Team Operating Model

- **CX:** UX build and interaction modeling
- **CC:** constitutional/system accuracy review
- **CW:** narrative and stress-test critique

This is not a design contest now. It is a single pipeline:
- build,
- constitutional review,
- narrative stress test,
- iterate.

## 12) Deliverable Format (Current Phase)

For each UX pass, provide:
1. A working HTML prototype (landing + mission board minimum)
2. A short "Corroborated vs Falsified" note for the design
3. A "Why this appears" behavior demonstration
4. A verification path demonstration
5. Known limitations disclosed plainly
